# 毕业设计管理系统 — 业务逻辑与数据流文档

## 一、系统概述

本系统是一个基于 JSP + Servlet + MySQL 的毕业设计管理系统，采用经典 MVC 架构。

**技术栈：**
- 后端：Java 8 + Servlet 4.0 + JSP
- 数据库：MySQL 8.3（JDBC 直连，无 ORM）
- 前端：Bootstrap 5.3.3 + 原生 JavaScript
- 构建：Maven，部署在 Tomcat（端口 8086）

**三种用户角色：**
- 管理员（admin）：管理用户和公告
- 教师（teacher）：发布课题、审核选题、评阅文档
- 学生（student）：选题、提交文档、查看成绩

## 二、系统架构分层

```
┌─────────────────────────────────────────────────────┐
│                   浏览器（JSP 页面）                    │
├─────────────────────────────────────────────────────┤
│              AuthFilter（权限过滤器）                   │
├─────────────────────────────────────────────────────┤
│              Controller 层（Servlet）                  │
├─────────────────────────────────────────────────────┤
│                DAO 层（数据访问）                       │
├─────────────────────────────────────────────────────┤
│             SQLHelper（JDBC 工具类）                   │
├─────────────────────────────────────────────────────┤
│                MySQL 数据库                           │
└─────────────────────────────────────────────────────┘
```

**请求处理链路：**
```
浏览器请求 → AuthFilter 鉴权 → Servlet 处理业务 → DAO 查数据库 → 返回数据 → JSP 渲染 → 响应浏览器
```

## 三、数据库设计（8 张表）

### 3.1 核心表关系图

```
users (用户表)
  │
  ├── 1:N ──→ topics (课题表)           [teacher_id → users.id]
  │              │
  │              └── 1:N ──→ topic_selections (选题表) [topic_id → topics.id]
  │                             │
  ├── 1:N ──────────────────────┘                     [student_id → users.id]
  │
  ├── 1:N ──→ documents (文档表)        [student_id → users.id]
  │                                      [reviewer_id → users.id]
  │
  ├── 1:N ──→ announcements (公告表)    [publisher_id → users.id]
  │
  └── 1:N ──→ operation_logs (操作日志)  [user_id → users.id]
```

### 3.2 各表字段说明

| 表名 | 核心字段 | 说明 |
|------|---------|------|
| users | id, username, password(MD5), role, real_name, student_no, department, status | 所有用户统一存储，role 区分角色 |
| topics | id, title, description, teacher_id, max_students, selected_count, status | status: open/closed |
| topic_selections | id, student_id, topic_id, status, apply_reason, review_comment | status: pending/approved/rejected/cancelled |
| documents | id, student_id, topic_id, doc_type, title, content, file_path, status, score, feedback | doc_type: proposal/midterm/final |
| announcements | id, title, content, publisher_id, is_top | is_top=1 表示置顶 |
| defense_schedules | id, student_id, defense_time, room, score | 答辩安排（预留） |
| operation_logs | id, user_id, action, target, detail | 操作审计 |
| messages | id, sender_id, receiver_id, title, content, is_read | 站内信（预留） |

## 四、核心业务流程详解

### 4.1 用户认证流程

```
┌──────────┐    POST /login.action     ┌─────────────────┐
│ login.jsp │ ─────────────────────────→│ LoginController  │
└──────────┘   username + password      └────────┬────────┘
                                                  │
                                          PasswordUtil.md5(password)
                                                  │
                                                  ▼
                                        ┌─────────────────┐
                                        │    UserDao       │
                                        │ findByUsername() │
                                        └────────┬────────┘
                                                  │ 比对 MD5 密码
                                                  ▼
                              ┌─────────────────────────────────────┐
                              │  验证成功？                           │
                              ├── 是 → session.setAttribute("loginUser", user)
                              │        → 根据 role 跳转对应 dashboard
                              └── 否 → 返回 login.jsp 显示错误信息
                              
角色跳转目标：
  admin   → /dashboard.jsp（显示管理员统计面板）
  teacher → /dashboard.jsp（显示教师工作面板）
  student → /dashboard.jsp（显示学生选题面板）
```

**AuthFilter 鉴权逻辑：**
```
每个请求 → AuthFilter 拦截
  │
  ├── 公开路径（/, /login.jsp, /login.action, /css/*, /js/*）→ 放行
  │
  ├── session 中无 loginUser → 重定向到 /login.jsp
  │
  └── session 中有 loginUser → 检查 URL 前缀是否匹配用户 role
        ├── /admin/*  且 role=admin   → 放行
        ├── /teacher/* 且 role=teacher → 放行
        ├── /student/* 且 role=student → 放行
        └── 不匹配 → 重定向到 /dashboard.jsp
```

### 4.2 选题业务流程（核心流程）

这是系统最核心的业务，涉及学生、教师、课题三方交互：

```
阶段一：教师发布课题
━━━━━━━━━━━━━━━━━━━
教师 → POST /teacher/topics.action?action=add
     → TeacherTopicController
     → TopicDao.add(topic)
     → topics 表插入记录（status=open, selected_count=0）

阶段二：学生浏览并申请选题
━━━━━━━━━━━━━━━━━━━━━━━━━
学生 → GET /student/topics.action
     → StudentTopicController
     → TopicDao.findOpen() 查询所有 status=open 的课题
     → 展示课题列表（含教师姓名、剩余名额）

学生 → POST /student/topics.action?action=apply
     → 业务校验：
       ├── 检查该学生是否已有 pending/approved 状态的选题
       │   └── 有 → 拒绝，提示"已有选题申请"
       └── 无 → SelectionDao.add(selection)
             → topic_selections 表插入（status=pending）

阶段三：教师审核选题
━━━━━━━━━━━━━━━━━━━
教师 → GET /teacher/selections.action
     → TeacherSelectionController
     → SelectionDao.findByTeacherId(teacherId) 查询待审核列表

教师 → POST /teacher/selections.action?action=approve
     → SelectionDao.updateStatus(id, "approved", comment)
     → TopicDao.incrementSelectedCount(topicId)
     → 若 selected_count >= max_students → 自动关闭课题(status=closed)

教师 → POST /teacher/selections.action?action=reject
     → SelectionDao.updateStatus(id, "rejected", comment)
```

**选题状态机：**
```
                  教师审批通过
  pending ─────────────────────→ approved
    │                                │
    │ 教师拒绝                        │ 学生获得提交文档权限
    ▼                                ▼
  rejected                     可提交 proposal/midterm/final
    │
    │ 学生可重新申请其他课题
    ▼
  (结束)
```

### 4.3 文档提交与评阅流程

```
前提条件：学生必须有一个 status=approved 的选题记录

阶段一：学生提交文档
━━━━━━━━━━━━━━━━━━━
学生 → GET /student/documents.action
     → StudentDocumentController
     → 校验：SelectionDao 查询该学生是否有 approved 的选题
       ├── 无 → 提示"请先完成选题"
       └── 有 → 展示文档提交页面

学生 → POST /student/documents.action?action=submit
     → 提交内容：doc_type(proposal/midterm/final) + title + content + file_path
     → DocumentDao.add(document)  status=submitted
     → documents 表插入记录

文档类型说明：
  proposal → 开题报告
  midterm  → 中期检查报告  
  final    → 毕业论文/设计终稿

阶段二：教师评阅文档
━━━━━━━━━━━━━━━━━━━
教师 → GET /teacher/documents.action
     → TeacherDocumentController
     → DocumentDao.findByTeacherId(teacherId) 查询所有待评阅文档

教师 → POST /teacher/documents.action?action=review
     → 提交：score(分数) + feedback(评语) + status(reviewed/rejected)
     → DocumentDao.updateReview(id, score, feedback, status, reviewerId)
     → documents 表更新 score, feedback, status, reviewer_id, review_time
```

**文档状态流转：**
```
  draft → submitted → reviewed (评阅完成，附带分数)
                   └→ rejected (打回重做，学生可重新提交)
```

### 4.4 管理员业务流程

```
用户管理：
━━━━━━━━
管理员 → GET /admin/users.action
       → AdminUserController
       → UserDao.findAll() / UserDao.findByRole(role)
       → 展示用户列表（支持按角色筛选）

管理员 → POST /admin/users.action?action=add
       → 密码 MD5 加密 → UserDao.add(user)
       → users 表插入

管理员 → POST /admin/users.action?action=edit
       → UserDao.update(user)

管理员 → POST /admin/users.action?action=delete
       → UserDao.delete(id)

公告管理：
━━━━━━━━
管理员 → POST /admin/announcements.action?action=add
       → AdminAnnouncementController
       → AnnouncementDao.add(announcement)
       → 所有角色的 dashboard 均可看到公告
```

## 五、完整数据流图（端到端）

### 5.1 学生视角的完整生命周期

```
登录 → 查看公告 → 浏览课题 → 申请选题 → 等待审核
                                              │
                              ┌────────────────┤
                              │ 被拒绝          │ 被通过
                              ▼                ▼
                         重新选题        提交开题报告(proposal)
                                              │
                                        教师评阅 ──→ 打回则重新提交
                                              │
                                              ▼ 通过
                                        提交中期报告(midterm)
                                              │
                                        教师评阅 ──→ 打回则重新提交
                                              │
                                              ▼ 通过
                                        提交终稿(final)
                                              │
                                        教师评阅打分
                                              │
                                              ▼
                                        查看最终成绩
```

### 5.2 教师视角的完整生命周期

```
登录 → 发布课题(可发布多个) → 等待学生申请
                                    │
                                    ▼
                              审核选题申请
                              ├── 通过 → selected_count++
                              │         → 满额则自动关闭课题
                              └── 拒绝 → 附带拒绝理由
                                    │
                                    ▼ (学生提交文档后)
                              评阅开题报告 → 打分/打回
                              评阅中期报告 → 打分/打回
                              评阅终稿     → 最终打分
```

## 七、Session 与状态管理

```
Session 结构：
┌─────────────────────────────────┐
│  Key: "loginUser"               │
│  Value: User 对象               │
│    ├── id                       │
│    ├── username                 │
│    ├── role (admin/teacher/student) │
│    ├── realName                 │
│    └── department               │
└─────────────────────────────────┘

生命周期：
  登录成功 → 创建 Session → 每次请求 AuthFilter 校验 → 登出时 invalidate()
  超时（默认30分钟无操作）→ 自动失效 → 下次请求重定向到登录页
```

## 九、业务规则约束总结

| 规则 | 说明 |
|------|------|
| 一个学生同时只能有一个有效选题 | pending 或 approved 状态算有效，rejected 不算 |
| 课题有名额限制 | selected_count >= max_students 时自动关闭 |
| 提交文档的前提 | 学生必须有 approved 状态的选题 |
| 教师只能管理自己的课题 | 查询和操作都限定 teacher_id = 当前登录教师 |
| 密码存储方式 | MD5 单向哈希，登录时比对哈希值 |
| 文档评阅权限 | 只有指导该学生的教师可以评阅其文档 |
